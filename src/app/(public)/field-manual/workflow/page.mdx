The cycle in [Quick Start](/field-manual/quick-start) was the round-trip view. This chapter goes deeper: what each skill does on disk, what knobs it accepts, and the phase model that ties them together. The precise contracts &mdash; argument tables, file owners, MCP tools &mdash; live in [Logistics](/field-manual/logistics).

## The cycle at a glance

The core pipeline is five skills:

`/infil` &rarr; `/warno` &rarr; `/opord` &rarr; `/splash` (loop) &rarr; `/exfil`

Plus three side branches:

- **`/raid`** &mdash; fresh-OP shortcut. Collapses `/warno` + `/opord` for small OPs where architecture review is overkill.
- **`/chop`** &rarr; **`/chopchop`** / **`/nochop`** &mdash; OP-splitting. Carve one OP into two when scope has ballooned.
- **`/recon`** &mdash; read-only side branch. Research the OP, talk through tradeoffs, and hand off to the next skill without touching disk.

Every skill in the pipeline ships you into a **phase** &mdash; `infil`, `warno`, `opord`, `splash`, `chop`. The phase name is the &ldquo;where are we&rdquo; signal: after `/warno` you&rsquo;re in the warno phase, and small follow-up edits (a single Warning Order tweak, a Concept of Operations rewrite) belong to that phase without needing to re-type `/warno` from scratch. Each skill enforces its own scope &mdash; if you ask for opord-shaped work in the warno phase, the agent routes you to `/opord` instead of doing it silently.

`/sitrep` is a sync, not a phase &mdash; it pushes local progress and posts a log without ending the current phase. `/exfil` ends the cycle entirely: sync, optional close, delete local infiled dir.

## /infil &mdash; pull the op local

`/infil {N}` fetches operation `{N}` from the server into the infiled directory &mdash; `.naholo/local/infiled/` inside the repo for team mode (`naholo init`), or `~/.naholo/covert-ops/{codename}/infiled/` for covert mode (`naholo covert init`, where each project gets its own codename folder under your home dir). Only one OP can be infiled at a time, so passing `{N}` while another OP is in flight errors &mdash; `/exfil` the current one first.

What lands on disk:

- `OPERATION.md` with `## SITUATION` seeded from the OP&rsquo;s logs and notes. `## MISSION` and `## EXECUTION` are absent &mdash; owned by `/warno` and `/opord` respectively.
- `TASKS.md` &mdash; empty until `/opord` writes the task list.
- `notes/*.md` &mdash; any free-form notes already on the OP.
- `LOGS.yml` &mdash; server-side log snapshot for reference.
- `.base/` &mdash; 3-way merge baseline owned by the sync commands.

`/infil` with **no args** is a refresh: 3-way merges the server state into your local copy without clobbering edits. Use this when a teammate added a note or a log on the same OP mid-cycle.

## /warno &mdash; write MISSION

`/warno` is the **direction-setting** phase. It appends `## MISSION` to `OPERATION.md` with three subsections:

- **Concept of Operations** &mdash; two or three sentences naming the chosen approach. The &ldquo;what&rdquo; and the &ldquo;why.&rdquo;
- **Warning Orders** &mdash; one bullet per architectural decision. The &ldquo;how,&rdquo; expanded one decision at a time.
- **Target Reference Points** &mdash; a curated list of files / folders / globs a fresh session needs to read. For agents, not users &mdash; TRP keeps a future `/opord` from re-walking the codebase from scratch.

`/warno` lands you in the **warno phase**. Inside the phase, MISSION tweaks don&rsquo;t need a fresh `/warno` invocation &mdash; just chat the change or hand-edit `OPERATION.md`; the agent applies the edit and fires a TIMELINE bullet.

The bar before you leave the warno phase: every Warning Order is clear to **you**, not just the agent. If a WO is fuzzy, resolve it &mdash; elaborate it into something concrete, or drop it if it does not earn its slot. If something nags at you that the agent has not filed, spit it out and let the agent capture it as a new WO. Iterating in the warno phase is cheap; shipping a half-understood MISSION is not.

Stay at the architecture and concept level. TASKS and shipped diffs are where details get reviewed &mdash; if you bury detail in WOs, you kill momentum without adding signal. A few load-bearing key decisions are unavoidable, but resist piling them on.

If CONOPS or the WO list keep growing past the size you can hold in your head, that is the signal the OP is doing the work of two. Stop adding WOs and run `/chop "..."` to draft a split.

Want to research or talk through it first? `/recon` drops the session into a read-only phase &mdash; pull files, ask questions, debate tradeoffs &mdash; then tell the agent to run `/warno` with the prompt you settled on. The discussion never touches disk; the decision lands when you hand off.

## /opord &mdash; cut MISSION into tasks

`/opord` is the **detail-cutting** phase. It reads the populated MISSION and writes `## EXECUTION`: one `### TASK N — Title` section per task. Each task carries:

- **Intent** &mdash; one or two sentences naming the success criterion `/splash` uses to know the task is done.
- **Scheme of Maneuver** (optional) &mdash; ASCII diagram, wireframe, or signature diff when the change is shape-y enough that prose beats a list.
- **Course of Action** &mdash; the atomic action list: `Add` / `Edit` / `Move` / `Delete` / `Run` / `Manual` steps.

Tasks are sized for one `/splash`: a single cohesive change a reviewer can read as one diff. If a task title needs &ldquo;and&rdquo; or a comma, split it.

`/opord` also lands a phase. Mid-cycle plan revisions &mdash; insert a task, split one in two, drop an unstarted one, retitle, rewrite a Course of Action &mdash; happen by re-invoking `/opord "freeform"` (or hand-editing EXECUTION + `TASKS.md`). Already-shipped tasks (those with an `#### After-Action Report` heading) are treated as **immutable by default**: the agent will not edit, renumber, or drop them on its own. The recommended path for a follow-up is a fresh TASK via `/opord` so the change gets its own review window. If you genuinely need to rewrite a shipped task &mdash; restoring a deleted file, undoing a botched ship &mdash; revert the change by hand and tell the agent to overwrite the AAR. The default keeps history honest; the override is yours when you need it.

If a revision you ask for would contradict a current Warning Order, the agent pushes back with a drafted `/warno "..."` prompt &mdash; the MISSION revision it thinks the change needs. From there:

- **Draft looks right.** Tell the agent &ldquo;run `/warno` and `/opord` for me.&rdquo; It revises MISSION, then re-cuts the affected tasks under the new Warning Orders. One handoff, two skills.
- **Draft needs tweaks.** Copy the prompt, edit it, and run `/warno "..."` yourself. After MISSION settles, follow up with `/opord "..."` so EXECUTION re-aligns with the new direction.

The chain is MISSION first, then EXECUTION.

## /splash &mdash; ship one task

`/splash` implements exactly one task per invocation. It reads the task&rsquo;s Intent + Scheme of Maneuver + Course of Action, writes the code, runs the project&rsquo;s format + typecheck checks, then appends an `#### After-Action Report` to the same task section. The checkbox flips in `TASKS.md`, the skill stops, and you review the AAR before deciding to ship the next task.

`/splash` defaults to the **first unchecked task** in `TASKS.md`. To target a specific task &mdash; including one you just inserted via `/opord` &mdash; pass the integer: `/splash 5`. Freeform args after the integer are implementation context: `/splash 5 "use the existing helper in src/utils/foo.ts"`. The canonical shape is `/splash N "prompt"`, but the agent also takes casual prompts &mdash; `/splash revise the previous task, rename that helper` &mdash; and picks the right TASK from context.

### How `Manual` COA works

While `/splash` is running the COA, `Manual: {action}` items break the loop and pause the skill. These are user-owned steps the agent will not execute on its own &mdash; database migrations, secrets to paste, browser verifications, anything the agent should not run itself. On each one, `/splash` asks for a confirmation, with three paths out:

- **Done.** You ran the step. `/splash` continues with the next COA item.
- **Defer.** You are skipping for now. The step lands verbatim in the AAR Notes as a `Deferred manual step: ...` line so nothing is silently dropped.
- **Report back.** Something did not go cleanly &mdash; paste the error, describe what happened, share the step&rsquo;s output. Pick the **Other** option on the confirmation, or just dismiss the prompt and reply directly. The agent reads the report and decides the next move: retry, route to a follow-up, or fold the failure into the AAR.

Done and Defer both ship the splash &mdash; the difference is only whether the step is recorded as a deferral. Report back hands control back to the agent to assess what happened first.

### After Action Report(AAR)

The AAR is the **plan-vs-shipped** record: the COA was the plan, the AAR says what diverged and why. Written at end of splash, appended to the task section under a `#### After-Action Report` heading, with two labels.

**Deviations** &mdash; anything different from the planned COA. Two sources:

- **Agent-initiated.** Mid-COA improvisation. Compile errors or test failures surface gaps in the plan &mdash; the agent adjusts to hit the task&rsquo;s Intent and records the change with a brief reason.
- **User-initiated.** A revision splash redirected scope (see Adjust fire below for the full revision flow).

**Notes** &mdash; follow-ups, deferred `Manual:` steps, risks worth flagging. Omitted when there is nothing to surface.

### Adjust fire

Every splash lands somewhere. Review the AAR + the diff. If it landed off-target, branch by size:

- **Minor.** You are still in the splash phase. Either chat the change &mdash; &ldquo;rename that helper,&rdquo; &ldquo;the AAR missed a step&rdquo; &mdash; and the agent fixes the code and refreshes the AAR. Or re-run `/splash N "tweak: ..."` &mdash; re-running on a shipped task is a **revision splash** that overwrites the AAR body in place under the same heading. The original Intent / SoM / COA stay untouched; those are `/opord`&rsquo;s planning record, immutable from a splash.
- **Major.** Re-invoke `/opord "..."` to insert a new TASK and ship the rework as its own splash. A new TASK gets a new review window; the original AAR stays as the historical record.

The bar: would a reviewer want this change as its own diff? Yes &rarr; new TASK. No &rarr; chat it in-phase.

## /sitrep &mdash; checkpoint sync

`/sitrep` is the mid-cycle push. It syncs the current state of local tasks and notes to the server, posts a short summary log, and leaves the local infiled directory intact for continued work. Optional freeform args become extra context for the summary log.

`/sitrep` does **not** end the current phase, so you can keep iterating after a checkpoint. Use it during long ops or multi-day work where teammates need visibility before `/exfil`.

## /exfil &mdash; close + clean up

`/exfil` ends the cycle. It runs the same sync as `/sitrep`, posts a summary log, optionally closes the operation server-side, and deletes the local infiled directory. Run when the OP is done or paused indefinitely and you want to free the single-infil slot for another OP.

`/exfil` decides whether to close the OP from `TASKS.md`: every task done &rarr; close; anything still open &rarr; pause and ask. Override the default with a freeform hint &mdash; `/exfil "close"` closes even with unfinished tasks, `/exfil "don't close"` keeps the OP open server-side even when every task shipped.

## OP-splitting side branch &mdash; the chop family

When scope has ballooned and the current OP is doing the work of two, split it. CHOP &mdash; Change of Operational Control, the military term for transferring a unit between commanders &mdash; is a proposal/apply flow.

**`/chop "freeform"`** drafts the split at `notes/CHOP.md`: a side-by-side brief with a `# CURRENT OP` block (what stays) and a `# NEW OP` block (what gets carved off). The freeform argument is **required** &mdash; it is the topic, Warning Orders, or new-OP title you want carved. Once the draft lands, you are in the **chop phase**: chat any revision and the agent edits `CHOP.md` in place. Re-invoke `/chop "..."` when a fresh session needs to re-enter the phase.

**`/chopchop`** applies: spawns the new OP server-side seeded with the carved SITUATION + MISSION + EXECUTION (including AARs for shipped tasks that were carried over), prunes the same scope from the parent&rsquo;s `OPERATION.md` + `TASKS.md`, and deletes `CHOP.md` both locally and server-side.

**`/nochop`** discards: deletes `CHOP.md` everywhere; neither OP is touched.

While `CHOP.md` exists, `/warno` / `/opord` / `/splash` each surface a &ldquo;CHOP pending&rdquo; gate before running &mdash; their edits would desync the proposal, so you have to decide whether to revise CHOP after (proceed) or resolve CHOP first (cancel and pick `/chopchop` / `/nochop` / `/chop`).

## /recon &mdash; read-only research and discussion

`/recon` drops the session into a read-only phase. It loads `OPERATION.md` for context, then answers questions, pulls extra files (`LOGS.yml`, other notes, codebase) on demand, and lets you talk through tradeoffs with the agent before committing to a phase-changing skill. When you have decided, tell the agent to run the next skill &mdash; `/warno "..."`, `/opord "..."` &mdash; and it picks up from the discussion. `/recon` itself writes **nothing**: no `OPERATION.md` edits, no `TASKS.md` edits, no TIMELINE bullets, no server syncs.

Runnable from any post-`/infil` state. Use it to research before a `/warno` revise, audit prior decisions, or get the lay of the land in a fresh session. The recon phase ends when a phase-changing skill (`/warno` / `/opord` / `/splash` / `/chop` / `/chopchop` / `/nochop`) or `/exfil` runs; `/sitrep` is sync-only and leaves recon intact.

If you ask the agent to silently do phase-shaped work mid-recon (e.g. &ldquo;rewrite MISSION to X&rdquo;), it routes you to invoke the right skill instead. The hand-off is explicit on purpose &mdash; recon&rsquo;s contract is that nothing lands on disk until you say so.
