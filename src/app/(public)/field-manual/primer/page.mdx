The agent is _always_ confident. However you run it (vibe coding, token-maxing, ship-and-see), the output is a slot machine: sometimes it works, sometimes it doesn&rsquo;t, and the confident tone never changes. The real problem isn&rsquo;t the misses. It&rsquo;s that the agent produces more output than any human can review, and three things break at once.

**The human breaks.** You end up in one of two corners. The optimist stops reading the output and trusts a confident voice until it bites; that is delusion. The conscientious one tries to keep up with every diff, every transcript, and every tool call until they drown; that is overload. Both are the same failure: no checkpoint is small enough to actually digest.

**The output breaks.** The devil hides in the details. Uncontrolled output stacks up, the next session learns from it, and more monkeypatched code piles on top. Complexity climbs, each implementation&rsquo;s diff bloats, and the agent improvises more and burns more tokens while the project drifts into chaos.

**The resource bill breaks.** Nobody counts the tokens, and the &ldquo;more context, more retries&rdquo; reflex pours compute, electricity, and water into the void. You can&rsquo;t track what this way of working hides.

These are three symptoms of one mistake: treating _more_ as a substitute for _review_.

## How you&rsquo;re supposed to use a coding agent

The cure is the same shape for every tool, whether it&rsquo;s Naholo, something else, or your own scripts. Three reviews carry every change, and if you skip one the others can&rsquo;t save you.

- **Review the architecture decisions** before any implementation is planned.
- **Review the implementation plan** before a single line of code is written.
- **Review the changes** before you trust them in front of anyone else.

Reviews are heavy, and the raw work of reviewing won&rsquo;t shrink much, so don&rsquo;t pretend it gets cheaper. What you can do is fold review into the act of shaping the idea, so each one becomes real value instead of a tax. Keep every review **small** and **cumulative**: small enough to hold in one sitting, and ordered so each builds on the last.

Start with the architecture. Let the agent research the codebase, but hold its output to the direction alone: a handful of decisions, not a sprawling plan doc. You review what the agent is thinking in a couple of minutes, and because there is no giant document, no tokens go to waste and your sanity stays intact.

Once the direction looks right, have the agent cut the work into tasks that each fit in a single commit. Every task stays concise and sticks to interfaces rather than exact code, unless a detail is genuinely critical. It covers two things: how the exposed interfaces change, shown concretely because that ripples out to other modules and other tasks, and how the internal procedure changes, shown conceptually, where a flow chart beats prose. That leaves a few symbol changes and a short conceptual summary you can read in about five minutes. Don&rsquo;t review every task up front, though, because course changes and pivots happen as you go. Tasks are written to build incrementally, so you review each one and ship it to confirm the change, and the implementation review takes minutes too.

None of this is novel. Plenty of people already work similar way, whether through their own skill docs or just by instinctively keeping each ask small so the agent doesn&rsquo;t run too far.

## How Naholo helps

Naholo holds the bound for you. An operation is pinned to the three reviews, one document per review:

- **WARNING ORDER** captures the architecture decisions so you can review them.
- **OPERATION ORDER** captures the implementation plan so you can review it.
- **The per-task After-Action Report** captures each change so you can review it in isolation.

A fourth document, **SITUATION**, sits in front of the trio. It isn&rsquo;t a review checkpoint; it&rsquo;s the agent&rsquo;s structured summary of your raw intel, the prep work that lets the three reviews land sharply.

Each review ends at a checkpoint. Each checkpoint is small enough that you don&rsquo;t drown in it, and structured enough that you can&rsquo;t skip it. That is the answer to the human cost: the load is chopped before it arrives at you.

Every operation also gets a token receipt. The agent-stats panel on the web app surfaces token usage by skill, model, and session so you can see what each operation actually cost. This way of working hides what you spend; Naholo shows it.

The skills (`/infil`, `/warno`, `/opord`, `/splash`, `/exfil`) and the CLI underneath them are the mechanisms that hold the shape in place.
