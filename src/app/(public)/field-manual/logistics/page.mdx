[Quick Start](/field-manual/quick-start) walks the happy path end to end. This chapter is the deep dive on what comes before: what you need on hand, what to install, which init mode fits, and what kind of agent tool the chain actually runs in.

## Prerequisites

Before you run `/infil` on your first operation, have these in place:

- **A naholo account and a project.** Sign up, create or join a project. Operations belong to projects.
- **A repo and an operation.** The repo is where the agent ships code. The operation is the issue you&rsquo;re about to work on — create it in the web app and drop logs as ideas land.
- **A coding agent.** [Claude Code](https://claude.com/claude-code) is the supported one today. Codex support is on the way.
- **The naholo CLI.** Installed once per machine, used by the agent during the cycle. You set it up here; you almost never type into it directly afterwards.

## Install the CLI

Install `naholo` globally from npm.

```bash
# with npm
npm install -g @naholo/cli

# or with pnpm
pnpm add -g @naholo/cli
```

The binary registers as `naholo` on your `PATH`. Confirm with `naholo --version`.

## Initialize

Logging in and wiring a repo are two steps.

### Log in

```bash
naholo login
```

Opens a browser for device-flow auth. Confirm the word pair shown in the CLI matches the one in the browser. On success, a user API token is saved to `~/.naholo/profiles/{name}.yml`. Run this once per machine.

### Pick an init mode

There are two ways to attach a repo to a project. Pick based on whether anyone else on the project will use naholo from this codebase.

**`naholo init` — team setup.**

```bash
# in the root of your repo
naholo init
```

Writes `.naholo/config.yml` (project id, slug, default operator id — **commit this**) and `.mcp.json` (usually gitignored). Anyone else who clones the repo can run `naholo login` and inherit the project wiring from the committed config.

**`naholo covert init` — solo setup.**

```bash
naholo covert init
```

Same effect as `naholo init` but the project wiring lives at `~/.naholo/covert-ops/` instead of in the repo. Nothing lands in the working tree, nothing gets committed. Pick this when you&rsquo;re the only person on the project who uses naholo and you don&rsquo;t want the config visible to teammates.

The caveat: covert config is per-working-directory. Each `git worktree` you create is a new working directory, so each worktree needs its own `naholo covert init`. Regular `naholo init` doesn&rsquo;t have this problem because the config is committed and travels with the worktree.

Either flow prompts to install the agent skills into `.claude/skills/` (default yes). If you decline now, run `naholo install-skills` later — same outcome.

## The agent tool

The chain — `/infil` → `/warno` → `/opord` → `/splash` → `/exfil` — only works in a tool that can **persistently open and edit `OPERATION.md`**.

Think of `OPERATION.md` as a cousin of the temporary plan documents agents already produce — &ldquo;plan mode&rdquo; output, inline planning blocks, scratchpads — but **durable**. It is a real file on disk that survives every checkpoint, `/clear`, and restart in between — until `/exfil` tears it down at the end of the cycle.

**Works:** an IDE with an agent plugin. [Claude Code](https://claude.com/claude-code) inside VS Code is the canonical setup for now — your agent runs in chat, your editor is right next to it, and `OPERATION.md` opens like any other file. You can edit it by hand, the agent can edit it in chat, and both of you see the same document.

**Doesn&rsquo;t work:** a pure chat interface (e.g. the standalone claude.ai desktop app). The agent there can produce planning text inside the chat, but that text is volatile — it scrolls away, it isn&rsquo;t a real file, and you can&rsquo;t hand-edit it. The cycle&rsquo;s phase reviews land on `OPERATION.md`, which has to be a durable artifact on disk. A chat-only surface can&rsquo;t hold one.

If you&rsquo;re wondering _why_ durability and operator control of the document matter so much, the [Primer](/field-manual/primer) is the chapter for that.
